# workspacecord `Discord` 状态消息与摘要消息底部跟随设计

日期：2026-04-02  
状态：已确认，待用户审阅  
范围：`Discord` 主会话频道中的状态消息与摘要消息展示行为调整

## 1. 背景

当前 `workspacecord` 在 `Discord` 主会话频道中有两类持续编辑的消息：

1. 状态消息
2. 最近摘要消息

现有实现把状态消息设计成常驻顶部消息，并在创建、编辑后持续尝试 `pin`。这在“频道是一个实时作战面板”时有可观察性优势，但在长对话场景下会带来明显体验问题：

- 用户继续聊了很多之后，状态与摘要仍然停留在频道最上方
- 想回看这两条消息时，需要反复滚回顶部
- 频道实际阅读重心已经转移到底部，但工作消息没有跟随最新对话位置

本次需求不是取消状态与摘要，而是改变它们的空间位置语义：

- 不再长期占据顶部
- 跟随新的用户轮次移动到底部
- 同一轮内部仍然保持“编辑现有消息”的低噪音行为

## 2. 目标与非目标

### 2.1 目标

1. 保留两条独立工作消息：状态消息、摘要消息
2. 用户发出一条新的入站消息后，把上一轮正在使用的工作消息迁移到频道底部
3. 一旦迁移完成，本轮内继续沿用现有逻辑，只编辑这两条最新消息
4. 移除“始终在顶部”的体验问题
5. 尽量复用现有状态机、摘要聚合、投递与消息编辑机制，降低改动面

### 2.2 非目标

1. 不把状态与摘要合并成一条消息
2. 不改成“每次状态变化都删除并重发”
3. 不把状态与摘要并入普通回复消息
4. 不在本轮每次摘要变化时触发重新建消息
5. 不在本次范围内重做整个实时面板架构
6. 不要求本次同时补齐摘要消息跨重启恢复能力

## 3. 用户确认后的目标体验

用户已确认采用以下方案：

- 状态消息与摘要消息保留为两条独立消息
- 同一轮内部继续编辑，不因为状态变化而重建
- 只有当用户发出新的消息时，才把当前工作消息移动到底部
- 移动到底部后，后续本轮状态变化与摘要变化继续编辑这两条新消息

等价地说：

> “每轮开始时换一组最新的状态/摘要消息；每轮进行中继续编辑，不要一直挂在最上面。”

## 4. 方案对比与结论

### 4.1 备选方案

#### 方案一：用户发送新消息时，重建底部工作消息

- 触发点：仅用户新消息到达时
- 行为：把当前状态消息与摘要消息在底部重建，并切换后续编辑绑定
- 本轮内：继续编辑，不再重建

#### 方案二：等本轮第一次状态变化时再迁移

- 优点：少一次可能无效的搬运
- 缺点：用户发完消息后，工作消息不会立刻出现在底部，体验不直接

#### 方案三：去掉 `pin`，但保留原消息位置

- 优点：实现最轻
- 缺点：消息仍然留在旧位置，无法满足“发完消息后就跟到底部”的核心诉求

### 4.2 结论

采用方案一。

原因：

1. 与用户描述完全一致
2. 触发时机稳定、可预测
3. 不需要把迁移逻辑掺进状态变更链路
4. 能保留现有“同一轮内只编辑”的低噪音特性

## 5. 产品规则

### 5.1 消息模型

每个主会话频道在任一时刻最多存在一组“当前工作消息”：

1. 当前状态消息
2. 当前摘要消息（可为空）

状态消息负责回答“现在是什么状态”；摘要消息负责回答“最近发生了什么”。两者继续保持职责分离。

### 5.2 触发时机

仅当收到新的用户入站消息，并且该消息将真正驱动一次新的会话处理时，才触发“工作消息下移”。

不在以下时机触发：

- 普通状态刷新
- 摘要刷新
- 工具事件
- 等待人工状态切换
- 结果输出

### 5.3 下移后的本轮行为

下移完成后：

- 状态消息的所有后续变化继续编辑新状态消息
- 摘要消息的所有后续变化继续编辑新摘要消息
- 不再因为本轮内部的状态变化而重建

### 5.4 无摘要场景

如果触发下移时当前并没有摘要消息：

- 只迁移状态消息
- 不强制生成空摘要消息
- 等摘要首次真正出现时，再在底部创建摘要消息并持续编辑

## 6. 架构与数据模型调整

## 6.1 状态消息

当前 `src/discord/status-card.ts` 的核心问题是自带顶部常驻语义：

- 新建后会尝试 `pin`
- 编辑后会再次尝试 `pin`

本次调整后，状态消息从“常驻状态卡”转为“当前轮次状态消息”：

1. 保留现有 `initialize()` / `update()` / `adopt()` / `getMessageId()` 主能力
2. 去掉自动 `pin` 行为
3. 新增“按当前内容在频道底部重建并切换绑定”的能力
4. 保留现有 `validate()` 约束，继续确保状态消息只表达短状态，不承载总结正文

建议新增接口（命名可调整）：

- `recreateAtBottom(state, data): Promise<{ oldMessageId?: string; newMessageId: string }>`

该接口职责：

1. 根据当前最新状态生成新消息
2. 把内部绑定切到新消息
3. 返回旧消息与新消息标识，供上层决定后续删除顺序

### 6.2 摘要消息

当前 `src/discord/summary-handler.ts` 已维护 `digestMessageIds`，可以编辑已有摘要，也能清理陈旧摘要分块。该结构适合作为“整组摘要消息迁移”的基础。

本次调整后建议：

1. 保留现有摘要分块与编辑逻辑
2. 新增“把当前摘要整组重建到底部并切换绑定”的能力
3. 如果摘要是多段，迁移时必须整组处理，而不是只迁首段
4. 继续允许“没有摘要时不创建摘要消息”

建议新增接口（命名可调整）：

- `relocateDigestToBottom(): Promise<{ oldMessageIds: string[]; newMessageIds: string[] }>`

该接口只处理“当前已有摘要”的迁移；如果当前没有摘要，直接返回空结果。

### 6.3 会话绑定

当前持久化层只记录 `statusCardMessageId`，摘要消息标识只保存在 `SummaryHandler` 内存中。

本次设计采用最小改动原则：

1. 继续持久化状态消息 `messageId`
2. 继续不持久化摘要消息 `messageId`
3. 不在本次需求中扩展摘要跨重启恢复能力

原因：

- 用户问题聚焦于“消息总在顶部”，而不是“重启后继续接管摘要消息”
- 当前功能可先在运行时完成主要体验修复
- 持久化摘要组 `messageId` 会放大改动面，不属于本次最小闭环

### 6.4 入口位置

迁移触发应放在新的用户消息入口，而不是状态链路内部。

推荐入口：`src/message-handler.ts`

推荐时机：

1. 消息通过权限校验
2. 消息通过限流校验
3. 会话未处于 `isGenerating`
4. 入站消息至少有正文或附件摘要之一
5. 在实际调用执行器前，触发一次“把当前工作消息迁移到底部”

这样可以保证：

- 只有真正开始新一轮时才迁移
- 不会因为空消息或被拒绝消息导致界面变化
- 不把展示逻辑耦合进状态机事件更新路径

## 7. 详细交互流程

### 7.1 新一轮开始前的迁移流程

当 `handleMessage()` 决定接受该消息并进入执行阶段时：

1. 读取当前会话绑定的状态消息与摘要消息
2. 如果当前存在状态消息，则在底部创建新的状态消息
3. 如果当前存在摘要消息，则在底部创建新的摘要消息组
4. 将会话组件绑定切到新状态消息与新摘要消息组
5. 最后删除旧状态消息与旧摘要消息组
6. 再进入后续 `executeSessionPrompt()` 与本轮状态刷新流程

### 7.2 本轮中途刷新流程

迁移完成后，本轮内继续采用当前行为：

- `updateSessionState()` 驱动状态消息编辑
- `flushDigest()` 驱动摘要消息编辑
- `sendTurnSummary()` / `sendTurnFailure()` / `sendEndingSummary()` 仍发结果类消息

### 7.3 删除顺序

删除顺序必须晚于新消息创建与新绑定生效。

固定顺序：

1. 新状态消息创建成功
2. 新摘要消息组创建成功（如果有）
3. 绑定切换成功
4. 再删除旧状态消息与旧摘要消息

设计原则：

> 宁可短暂重复，也不能出现“当前工作消息瞬间消失”。

## 8. 失败处理与降级策略

### 8.1 新状态消息创建失败

如果新状态消息创建失败：

- 整次迁移取消
- 继续沿用原状态消息与原摘要消息
- 本轮继续正常执行
- 记录警告日志，便于定位

原因：状态消息是当前工作消息组的主锚点；没有它，不应做半迁移。

### 8.2 新状态消息创建成功、摘要迁移失败

如果：

- 新状态消息创建成功
- 新摘要消息组迁移失败

则：

1. 切换状态绑定到新状态消息
2. 旧摘要消息暂时保留，不删除
3. 摘要后续第一次刷新时，再尝试以新绑定方式补建
4. 补建成功后，再清理旧摘要消息

这样可以避免“状态下来了，摘要消失了”。

### 8.3 删除旧消息失败

如果新消息已创建并切换绑定成功，但删除旧消息失败：

- 不回滚新绑定
- 本轮继续使用新消息
- 旧消息残留允许短期存在
- 记录警告日志
- 下次迁移时可再次尝试顺手清理

### 8.4 无摘要消息时

如果本次迁移时没有旧摘要消息：

- 视为正常场景
- 只迁移状态消息
- 不产生日志告警

## 9. 对现有模块的建议改动

### 9.1 `src/discord/status-card.ts`

建议改动：

1. 去掉 `sendNewMessage()` 与 `editExistingMessage()` 内部的自动 `pin`
2. 提供获取当前可复用渲染参数或直接按参数重建的能力
3. 提供底部重建接口
4. 保持现有校验逻辑不变

### 9.2 `src/discord/summary-handler.ts`

建议改动：

1. 保留 `digestMessageIds`
2. 保留现有分块与编辑逻辑
3. 增加迁移已有摘要组到底部的能力
4. 明确区分“摘要迁移”和“摘要刷新”两个路径

### 9.3 `src/panel-adapter.ts`

建议改动：

1. 对外暴露“开始新一轮前迁移工作消息”的统一入口
2. 由该入口协调状态消息迁移与摘要消息迁移
3. 迁移成功后同步更新会话绑定

建议新增接口（命名可调整）：

- `relocateSessionPanelToBottom(sessionId): Promise<void>`

由它负责：

- 查找当前组件
- 迁移状态消息
- 迁移摘要消息
- 更新 `sessions.setStatusCardBinding()`
- 处理旧消息删除顺序与失败兜底

### 9.4 `src/message-handler.ts`

建议改动：

在 `updateSession(session.id, { lastInboundMessageId: message.id })` 与 `executeSessionPrompt(...)` 之间插入：

- `await relocateSessionPanelToBottom(session.id)`

前提是：

- 会话是主会话或允许该能力的线程会话
- 当前会话组件已初始化
- 本条消息将真正进入执行

## 10. 测试策略

### 10.1 单元测试

#### 状态消息

新增或调整测试，覆盖：

1. 新建状态消息后不再调用 `pin`
2. 编辑状态消息后不再调用 `pin`
3. 底部重建成功后 `messageId` 切换到新消息
4. 底部重建失败时保留旧 `messageId`

#### 摘要消息

新增或调整测试，覆盖：

1. 已有单段摘要可迁移到底部
2. 已有多段摘要会整组迁移到底部
3. 迁移后 `digestMessageIds` 切换到新组
4. 迁移失败时保持旧摘要组不丢失
5. 无摘要时迁移接口安全返回

#### 统一入口

在 `panel-adapter` 侧覆盖：

1. 只迁移状态，不迁移摘要
2. 状态成功、摘要失败的半降级路径
3. 新绑定成功后删除旧消息
4. 删除旧消息失败不影响新绑定

#### 消息入口

在 `message-handler` 侧覆盖：

1. 新用户消息进入执行前会触发一次迁移
2. 被拒绝、限流、空消息等场景不会触发迁移
3. 迁移失败不会阻断 `executeSessionPrompt()`

### 10.2 回归测试

重点确认以下旧行为保持不变：

1. 同一轮内状态变化仍然只是编辑
2. 同一轮内摘要刷新仍然只是编辑
3. 本轮总结、失败总结、结束总结仍然发送新结果消息
4. 等待人工交互卡逻辑不受影响

## 11. 风险与取舍

### 11.1 频道底部噪音略增

每轮开始都会新建一组工作消息，理论上频道消息数会多于“永远编辑同一条”的模式。

接受该取舍，因为用户明确更看重“跟着当前对话走”，而不是“顶部常驻”。

### 11.2 摘要跨重启恢复仍不完整

本次不补摘要组持久化，因此进程重启后只能继续接管状态消息，不能继续接管上一轮摘要组。

接受该取舍，因为这不是当前痛点。

### 11.3 迁移时短暂重复

采用“先建新、后删旧”后，频道在极短时间内可能同时存在旧工作消息与新工作消息。

接受该取舍，因为它优于“删早了导致当前工作消息空窗”。

## 12. 验收标准

1. 状态消息不再在创建或编辑后自动 `pin`
2. 用户发送新的有效消息后，当前状态消息会在频道底部重建
3. 若存在摘要消息，摘要消息组也会在频道底部重建
4. 新绑定建立后，旧状态消息与旧摘要消息会被删除；删除失败不影响本轮继续运行
5. 同一轮内的状态变化与摘要变化继续编辑当前最新那组消息，不重复重建
6. 被拒绝消息、空消息、限流消息不会触发工作消息迁移
7. 结果类消息发送行为保持不变

## 13. 推荐实施顺序

1. 先改 `StatusCard`，移除 `pin` 并支持底部重建
2. 再改 `SummaryHandler`，支持摘要组底部迁移
3. 在 `panel-adapter` 增加统一迁移入口
4. 最后在 `message-handler` 接入触发点并补测试

## 14. 后续可选扩展

1. 持久化摘要组 `messageId`，支持重启后继续接管摘要
2. 为旧残留工作消息增加低频清扫机制
3. 允许按配置切换“顶部常驻模式”与“底部跟随模式”
4. 在多线程会话里细分是否启用该体验
